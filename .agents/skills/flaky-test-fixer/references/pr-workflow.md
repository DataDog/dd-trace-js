# PR Workflow

This document covers the mechanics of turning a verified fix into a draft PR. It encodes the conventions used for flaky-test fixes in this repo. Follow it exactly — these conventions are stable across the PRs that have landed under this skill.

## Before you touch git

A fix is not ready to commit until **all** of these are true:

1. The root cause has been identified mechanistically (write the one-sentence cause down somewhere).
2. The fix has been applied and is scoped to that one cause. No drive-by changes.
3. The test fails without the fix (verified by reverting locally) — required for newly added regression tests.
4. The test passes with the fix.
5. A 20-50x local loop is clean.
6. The whole spec file passes (not just the grep'd test).

If any of these isn't true, stay in the verification loop.

## Branch naming

`<your-prefix>/fix-<area>-<short-symptom>` — kebab-case throughout. Most contributors use their GitHub handle as the prefix; check what prefix the user normally uses (e.g. `git branch | head -20`) and match it. If there is no clear convention, ask.

Example shapes:

- `<handle>/fix-http2-port-reuse`
- `<handle>/fix-http2-cancelled-request-span`
- `<handle>/fix-electron-ipc-tests`
- `<handle>/fix-couchbase-readiness`
- `<handle>/fix-mongoose-overwrite-model`
- `<handle>/fix-rasp-metrics-preload`

Keep the symptom concise — the branch name is informational, the title and body carry the detail.

## Commit message

**Conventional Commits with a precise scope.** The type encodes whether production code changed:

- `test(<scope>):` — change is contained to `test/`, `integration-tests/`, or `packages/datadog-plugin-*/test/`.
- `fix(<scope>):` — production code under `src/` changes (even if the bug was infrastructural).
- `ci(<scope>):` — change is in `.github/workflows/`, Docker, or service-readiness scripts.
- `chore(<scope>):` — change is purely supporting (lint, dependency, dev tooling).

The scope is usually the plugin / area: `http2`, `electron`, `aws-sdk`, `mongoose`, `appsec`, `debugger`, `profiling`, `couchbase`, `kafkajs`. Use the same scope as the affected package directory when possible.

Examples (titles only, drawn from merged PRs):

- `test(http2): fix flaky cancelled-request span assertion`
- `test(http2): avoid port reuse in server tests`
- `fix(electron): fix flaky IPC tests and Xvfb startup race`
- `fix(google-cloud-pubsub): poll for GC collection instead of fixed retry count`
- `fix(mongoose): make PeerCat model registration idempotent`
- `test(aerospike,couchbase): fix hook timeouts and couchbase service readiness`
- `fix(appsec): increase beforeEach timeout in rasp-metrics integration test`

Title is short (under 70 chars). Body explains the *why*, not the *what* — the diff shows the what. For non-trivial PRs (e.g. the wall-clock-vs-monotonic one), the body should be a short paragraph naming the mechanism, not a bulleted list.

End the commit message with the standard Co-Authored-By line, using the actual model handling the work — *do not impersonate a different model*. The line format is:

```
Co-Authored-By: Claude <handle> (1M context) <noreply@anthropic.com>
```

…where `<handle>` is your model name (e.g. `Opus 4.7`, `Sonnet 4.6`).

## PR body template

Use this skeleton. Adapt section headings if the change merits a `### Motivation` block (e.g. the debugger sampling PR), but keep the basic structure.

```markdown
## Summary

- <One bullet per logical change. Each bullet says *why*, not just *what*. Two sentences max.>
- <If the change is non-obvious, briefly cite the mechanism (race, clock skew, ordering, etc).>

## Test plan

- [ ] <Plugin command if applicable, e.g. `PLUGINS="http2" npm run test:plugins`>
- [ ] <Loop run if the flake required one to detect, e.g. "ran the failing spec 50x locally with no failure">
- [ ] <For new regression tests: "confirmed test fails when the fix is reverted">

> Generated by [Claude Code](https://claude.com/claude-code)
```

**Do NOT include a `## Semver` section.** Semver-patch / minor / major status is set via PR label, not in the body.

For PRs that merit a longer narrative (clock-skew, two-process races, complex root causes), expand into `### Motivation` after the Summary. Use the `test(debugger): widen sampling spec tolerance to absorb wall-clock skew` PR body as the reference template — it walks through the mechanism, names the production code paths involved, and explains why the production code is *intentionally* the way it is.

## The commit/push checkpoint protocol

**Never commit or push without explicit user confirmation.** This applies at every stage:

- After the fix is implemented and verified → show the diff with `git diff` or `git status`, propose a commit message, **ask** before running `git commit`.
- After commit → **ask** before running `git push`.
- After push, even on a branch with an open PR → **ask** before any subsequent commit or push.

Acceptable phrasings:

- "The fix is verified — staged diff below. Want me to commit with `test(http2): fix flaky cancelled-request span assertion`?"
- "Commit is on the branch. Push?"

Do not assume "yes" from "looks good" or similar. Wait for an explicit go.

## Opening the PR

Always open as a draft:

```bash
gh pr create --draft --title "<conventional commit title>" --body "$(cat <<'EOF'
## Summary

- <bullet>

## Test plan

- [ ] <validation step>

> Generated by [Claude Code](https://claude.com/claude-code)
EOF
)"
```

After the PR is open, share the URL with the user. They will mark it ready-for-review themselves.

## Commenting on a PR

If the user asks you to comment on an existing PR (yours or someone else's), the comment body must say it was generated by Claude Code:

```bash
gh pr comment <num> --body "$(cat <<'EOF'
<your comment>

> Comment generated by Claude Code
EOF
)"
```

This applies to issue comments and review comments alike.

## After feedback

When the user requests changes on the PR (via review comments or chat):

- **Always add new commits.** Do not rebase or squash during review — keep history additive so reviewers can re-read just the deltas.
- Same commit-message format. Subsequent commits can be smaller (`fixup: addr review note on N1QL probe path`), but keep them Conventional Commits compliant.
- Same checkpoint protocol — ask before each commit, ask before each push.

If the user explicitly asks for a force push (rare), warn them about the destination branch and confirm intent before running it. Never force-push to `master`.

## Lint and validation

Before declaring the PR ready, run lint:

```bash
npm run lint
```

If it fails, fix the lint issues in the same PR before pushing again. Don't open a separate lint-cleanup PR for code you just touched.

If the project has additional validation scripts called out in CLAUDE.md for the affected area, run them too.
