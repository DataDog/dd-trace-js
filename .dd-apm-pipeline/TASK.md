# Task: Validate Current Vercel AI SDK Support

## Confirmed Starting Point

- This branch starts from `dd-trace-js` `origin/master` at `51fdd86fea6cacaec820cdac4158904bda691511`.
- Vercel AI SDK v7 support already exists from PR #8949 / commit
  `c3db8f8b1032ee4e0cf73a81d788ac22820a9d65`.
- Instrumentation declares `ai >=7.0.0`, and dedicated APM and LLMObs v7 suites exist.
- At bundle creation on 2026-07-15, npm reports `ai@7.0.22` as latest, while
  `packages/dd-trace/test/plugins/versions/package.json` caps the tested version at `7.0.2`.

## Goal

Advance dd-trace-js's verified Vercel AI SDK ceiling to the latest stable v7 release available when
the task runs. Prove that both APM and LLMObs behavior still work with compatible current provider
packages. Fix production instrumentation only when a concrete latest-version reproduction fails.

If current v7 works without production changes, the correct result is a narrow test-matrix and lockfile
update plus evidence. Do not invent compatibility code to make the diff look more substantial.

## Required Workflow

1. Use a read-only subagent to inspect PR #8949, the v7 instrumentation paths, version harness, and
   relevant APM/LLMObs tests. It must identify the smallest expected diff and likely compatibility risks.
2. Confirm current stable versions from the npm registry. Record exact versions and publication metadata.
3. Update the version harness ceiling and compatible provider pins needed by the existing v7 suites.
   Regenerate repository-managed dependency artifacts using established dd-trace-js commands.
4. Use focused subagents for implementation, test diagnosis/fixing, and final review. The main agent
   coordinates, reviews diffs, maintains `PROGRESS.md`, and enforces gates.
5. Run the real plugin and LLMObs AI suites against the selected versions. A failing latest-version test
   is the required reproduction before changing production code.
6. Run a minimal real sample app through the tracer and capture actual APM and LLMObs output. Exercise
   non-streaming generation, streaming generation, embeddings, and a tool call.
7. Complete every final gate in order. Any source change invalidates prior final-gate receipts.

## Scope

Expected first-line scope:

- `packages/dd-trace/test/plugins/versions/package.json`
- dependency lock/workspace artifacts generated by the repository's normal version-install process
- existing `datadog-plugin-ai` and LLMObs AI tests only when coverage must change for a demonstrated gap

Production instrumentation files are out of scope unless tests or the sample app reproduce incorrect
behavior on the selected latest release. Avoid unrelated provider upgrades, refactors, generated evidence,
or broad test cleanup.

## Acceptance Gates

- The checked-in version ceiling matches the selected current stable `ai` v7 release.
- Existing v7 APM and LLMObs suites run against that release and compatible provider versions.
- Build, targeted tests, and lint pass from one unchanged source revision.
- A captured APM trace contains the expected AI SDK operations, including generation, step/model call,
  embedding, and tool execution where emitted by the SDK.
- A captured LLMObs trace has correct model/provider data, token metrics, inputs/outputs, hierarchy,
  streaming completion, and no duplicate terminal spans.
- Evidence summaries are bounded and reviewable; raw traces/logs stay under gitignored
  `.dd-apm-pipeline/evidence/*/raw/` and are never added to the product commit.
- A separate review subagent finds no unresolved correctness, compatibility, or maintainability issue.

## Stop Conditions

- If registry/network access prevents resolving the current versions, mark the relevant stage BLOCKED.
- If real trace capture infrastructure is unavailable, mark final observability BLOCKED rather than
  substituting unit tests.
- Do not open a PR while any final gate is failed, blocked, stale, or unreviewed.
